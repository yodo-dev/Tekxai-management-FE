# Desktop Agent CI/CD Restoration Plan

**Status: PROPOSAL ONLY — not implemented.** This document describes the design; implementation requires separate, explicit approval, since it involves creating new GitHub secrets and a workflow with production deployment access.

## Why this is needed

Confirmed this sprint: no CI/CD pipeline currently builds or deploys the desktop app. A prior pipeline (`build-desktop.yml`) existed from 2026-06-26 to 2026-06-30, then was deleted in commit `21bcc14` ("update action deployment") without a documented replacement. Since then, every release — including `desktop-v1.0.0` — has been a manual local build + manual file transfer. This sprint's deployment also surfaced a concrete failure mode of manual process: a wrong destination path (`~/downloads/` instead of `~/downloads/latest/`) silently shipped a stale installer while the metadata endpoint reported success. Automation removes this entire class of error.

## Build

- **Trigger**: push to `main` touching `desktop-app/**`, plus `workflow_dispatch` for manual runs
- **Windows build**: `windows-latest` GitHub-hosted runner, `npm ci && npm run build:win`
- **macOS build (optional)**: `macos-latest` runner, `npm run build:mac`, gated behind a workflow input so Windows-only releases don't wait on it
- **Version injection**: read `version` from `desktop-app/package.json` as the single source of truth; fail the build if it doesn't match the pushed git tag (when triggered by a tag) to prevent version drift
- **Build metadata generation**: capture `git rev-parse HEAD`, `git log -1 --format=%s`, `git log -1 --format=%an`, build timestamp (`date -u`), and computed SHA256 as workflow outputs, passed to the Release and Deployment jobs

## Release

- **Automatic Git tags**: on merge to `main` with a version bump in `package.json`, auto-create `desktop-v<version>` (or require an explicit tag push to trigger a release build — recommend the latter, since it keeps releases intentional rather than automatic on every merge)
- **GitHub Releases**: create a GitHub Release for each tag, attaching the `.exe` (and `.dmg` if built) as release assets — this also solves the "no way to relay a large binary" problem this sprint hit, since Release assets are the correct mechanism for binary distribution (vs. this sprint's CloudShell workaround)
- **Release notes generation**: auto-populate from commit messages since the last tag (`git log <last-tag>..HEAD --oneline`), with a manual free-text section for QA verification summary
- **Artifact storage**: GitHub Release assets serve as the durable, versioned artifact store — every past release stays downloadable and hash-verifiable independent of the production server's current state

## Deployment

- **Automatic upload to production**: SCP the built installer to `/home/ubuntu/downloads/latest/` on the backend EC2 host — using the **correct path this time**, hardcoded and covered by a deploy-time assertion, not just implied by convention
- **Automatic `metadata.json` generation**: generated by the workflow itself (version, commit, build timestamp, checksums, release notes) — never hand-written, closing the exact provenance gap found this sprint
- **Automatic SHA256 generation**: computed in the Build job, verified again post-upload (see Verification below) — never manually typed
- **Backup of previous release**: before overwriting, the deploy step copies the current `TekXAI-Agent-Setup.exe` and `metadata.json` to timestamped `.bak-<ISO8601>` files (same convention already used manually this sprint) — keep the last N (e.g. 5) backups, prune older ones
- **Atomic deployment**: upload to a temporary path (e.g. `TekXAI-Agent-Setup.exe.uploading`) first, verify its checksum, then `mv` (atomic on the same filesystem) into place — this guarantees no window where nginx could serve a half-written file, and prevents exactly the kind of "metadata updated, binary didn't" split-state this sprint hit

## Verification

- **Verify uploaded checksum**: immediately after the atomic move, SSH back and `sha256sum` the live file, compare against the Build job's recorded checksum — fail the workflow loudly if they don't match
- **Verify metadata**: `curl` the live `/api/v1/downloads/latest` endpoint from the runner (public internet, same path a real customer takes) and assert the returned `commit` field matches the commit being deployed
- **Verify downloaded installer**: `curl`-download the `.exe` from the public URL (not just check the file on-server) and SHA256 it — catches nginx config issues, caching layers, or DNS problems the on-server check alone would miss
- **Verify app.asar hash**: extract the asar from the downloaded installer (same `asar extract` approach used manually this sprint) and diff its hash against the one computed from the Build job's `win-unpacked` output — this is the strongest guarantee that "what got deployed" equals "what got built," closing this sprint's core verification gap permanently
- **Smoke test**: on a Windows runner (or self-hosted Windows runner, since GitHub-hosted Windows runners can run GUI apps headlessly via tools like `xvfb`-equivalents are not needed on Windows, but Electron apps can be launched with `--headless`-adjacent flags or driven via Playwright/Spectron-style Electron testing) — launch the installed app, assert the login screen renders, assert no crash within N seconds. Full interactive smoke testing (actual login) requires a test account's credentials stored as a GitHub secret — flag this as a decision point, not assumed.

## Rollback

- **One-command rollback**: a separate `workflow_dispatch`-triggered job (`rollback.yml` or a job within the main workflow) that takes a target timestamp/tag as input
- **Restore previous installer**: `cp` the matching `.bak-<TIMESTAMP>` file back over the live one, atomically (same temp-file-then-move pattern as deployment)
- **Restore metadata**: same pattern for `metadata.json.bak-<TIMESTAMP>`
- **Rollback verification**: re-run the exact same Verification steps above (checksum, metadata, downloaded installer, app.asar hash) against the rolled-back state — a rollback isn't "done" until it passes the same bar as a forward deployment

## Security

- **Secret management**: GitHub encrypted secrets (`BACKEND_SERVER_HOST`, `BACKEND_SERVER_USER`, `BACKEND_SERVER_SSH_KEY`) — never committed, never logged (GitHub Actions automatically masks secret values in logs, but explicitly avoid `echo`-ing them regardless)
- **SSH key handling**: recommend a **dedicated deploy key** scoped to this one purpose (SSH access for the deploy user, restricted via `authorized_keys` `command=` forced-command if possible, limiting it to only the specific rsync/scp operation needed) rather than reusing a general-purpose admin key. Rotate on a defined schedule (e.g. every 90 days) or immediately if this repo's secret access is ever suspected compromised.
- **Least-privilege deployment**: the deploy user on the EC2 host should only have write access to `/home/ubuntu/downloads/`, not broader sudo/root access — verify this is already true (the manual deployment this sprint used the `ubuntu` user directly, which likely has broader privileges than ideal; tightening this is a legitimate follow-up, tracked as backlog below rather than done silently)
- **Audit logging**: every deployment run's inputs (commit, actor, timestamp) and outputs (checksums, verification results) logged as workflow output/summary, retained per GitHub's standard Actions log retention — sufficient for "who deployed what when" without needing a separate logging system

## Notifications

- **GitHub deployment status**: use GitHub's native Deployments API (via `actions/github-script` or a dedicated action) so deployments show up in the repo's "Environments" view with success/failure state
- **Success/failure notifications**: at minimum, GitHub's own PR/commit status checks; optionally extend to Slack/email via a webhook step if the team already has a notification channel (don't invent a new one without confirming preference)
- **Deployment logs**: the workflow run's own log is the primary record; consider appending a one-line summary to `RELEASE.md` or a `CHANGELOG.md` automatically as a final step, so the deployment history is visible in-repo without needing GitHub Actions access

## Open decisions requiring your input before implementation

1. Tag-triggered releases (explicit, intentional) vs. auto-tag-on-merge (more automatic, more risk of accidental releases) — recommend tag-triggered
2. Whether to build macOS by default or gate it behind an input (macOS builds cost more CI minutes and aren't currently distributed based on this sprint's findings — recommend gating)
3. Whether smoke testing should include a real login (requires a test-account secret) or stop at "app launches without crashing"
4. Whether to tighten the deploy user's permissions on the EC2 host, and whether that's in scope for this restoration or a separate hardening task
